System architecture · repos · contracts

Three repos, one product.

QPay ships as a Flutter mobile app backed by a TypeScript API. Strategy and docs live separately. Here's why three repositories (not one monorepo), how they talk to each other, and what goes where.

Decision

Polyrepo, not monorepo.

Flutter (Dart) and Node (TypeScript) are different ecosystems with incompatible tooling. Mixing them in one repo creates more friction than it saves. We split by runtime boundary and wire them through a versioned HTTP contract.

How they talk to each other

A single HTTP REST API defined by OpenAPI schema. The schema lives in qpay-backend; a CI job publishes it as a versioned artifact and generates a Dart client package that qpay-app consumes. No untyped calls, no contract drift.

Rule of thumb: the mobile app never touches an external service directly. Stripe tokens, Companies House submissions, KYC webhooks — all go through backend first. Keeps secrets off the device and gives us a single audit log.

Why not a monorepo?

❌ Tooling mismatch

Dart uses pub, Node uses npm / pnpm. They don't share lockfiles, workspaces, or dependency resolution. A monorepo means running two separate package managers side by side — all the downsides of splitting, none of the upsides.

❌ CI times diverge

Flutter build + iOS/Android signing runs 15–20 minutes. Backend unit tests run in 2–3. In one repo every trivial backend PR waits on a mobile build. Separate repos keep each loop fast.

❌ Different secrets

App Store signing keys, Play Console credentials, provisioning profiles live in qpay-app. Database URLs, Stripe secret keys, Companies House passwords live in qpay-backend. Keeping them apart reduces the blast radius if either repo is compromised.

❌ Different contributors

Mobile contractors, backend contractors, accounting partners may all be granted access in different combinations. Separate repos make permission boundaries clear; a monorepo forces "all or nothing".

✅ Shared types — solved by OpenAPI

The one real monorepo win is "change a model on both sides at once". OpenAPI solves this without the monorepo cost: backend defines the schema, CI generates a typed Dart client, mobile imports it. Drift becomes a compile error, not a production bug.

✅ Atomic cross-stack changes — not needed at MVP

The mobile app and backend ship on different cadences anyway (App Store review = days, backend deploy = minutes). API versioning + backward compatibility is the correct pattern for mobile, not atomic commits.

Contract: OpenAPI schema

The schema is the single source of truth for every endpoint the mobile app can call. Defined in qpay-backend/openapi.yaml, validated in CI, published as a versioned artifact.

The pipeline

qpay-backend                                   qpay-app
  ├── openapi.yaml           ────┐
  ├── CI: validate schema        │
  │     generate Dart client     │
  │     publish to GH Packages   │
  │                              ▼
  │                         pubspec.yaml:
  │                           qpay_api: ^1.4.0
  │                              │
  │                              ▼
  │                         Dart imports
  │                         typed methods
  │                         auto-completed in IDE
  └── Runtime: serves endpoints       ⇅
                                  HTTPS (typed)

Every endpoint change goes through a backend PR that bumps openapi.yaml. CI generates a new Dart package version. Mobile upgrades in its own PR when ready. Backward-compatible changes ship freely; breaking changes bump the major version and coexist during rollout.
